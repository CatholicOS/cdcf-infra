## Recommended env vars per repo

**`LiturgicalCalendarAPI`** (PHP backend — JWT validation, OpenFGA checks):

```bash
ZITADEL_ISSUER=https://auth.catholicdigitalcommons.org
ZITADEL_PROJECT_ID=373246750732845059
ZITADEL_CLIENT_ID=373246751403933699        # ← the API Backend client_id (audience)
# ZITADEL_MACHINE_TOKEN=<out-of-band>       # only if making outbound Management API calls

OPENFGA_API_URL=https://authz.catholicdigitalcommons.org
OPENFGA_STORE_ID=01KRSCF4GVX0X4ZNXXJQEC4XXJ
OPENFGA_MODEL_ID=01KRSCF4K9W2EWZ1X2PP1QVH3B
# OPENFGA_PRESHARED_KEY=<out-of-band>
```

**`LiturgicalCalendarFrontend`** (PHP server-side — initiates PKCE login):

```bash
ZITADEL_ISSUER=https://auth.catholicdigitalcommons.org
ZITADEL_CLIENT_ID=373289176235245570        # ← the Frontend client_id (DIFFERENT from API's!)
# No client secret — PKCE flow handles auth without one.
# No OPENFGA_* vars here — frontend doesn't talk to OpenFGA directly.
# No ZITADEL_MACHINE_TOKEN — frontend only ever holds user tokens.
```

⚠ **Common mistake:** putting the API Backend's `ZITADEL_CLIENT_ID` in the frontend's env. The frontend's PKCE login flow will fail because the API app uses `PRIVATE_KEY_JWT` auth, not the PKCE/authorization_code flow. **They are two distinct apps with distinct client_ids; do not share them.**

## How to verify on LitCal's side

```bash
# OIDC discovery

# OpenFGA store ping

  https://authz.catholicdigitalcommons.org/stores/01KRSCF4GVX0X4ZNXXJQEC4XXJ
```
